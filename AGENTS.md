## POLÍTICA MAESTRA DE IDENTIDAD NOMINAL

### NOMBRE RECTOR OFICIAL

El nombre rector oficial del sistema es:

**OpenCore**

Este nombre gobierna la identidad conceptual, operativa y estructural del proyecto.

### ALCANCE DEL NOMBRE RECTOR

OpenCore debe usarse como nombre principal en:

- documentación nueva
- directrices operativas
- instrucciones para Codex
- narrativa del sistema
- definición de arquitectura
- decisiones estructurales
- nuevos módulos, flujos o capas cuando no exista restricción técnica

### NOMBRES HEREDADOS O SECUNDARIOS

Cualquier nombre alterno existente, incluyendo pero no limitado a:

- OperaCore
- AmeriCore

debe considerarse uno de estos casos:

1. nombre heredado
2. nombre técnico temporal
3. módulo secundario
4. referencia legacy pendiente de consolidación

Estos nombres no deben competir con OpenCore como identidad principal del sistema.

### REGLA DE CONSOLIDACIÓN

Toda nueva pieza creada a partir de este momento debe alinearse nominalmente con OpenCore, salvo que exista una razón técnica, histórica o contractual explícita para no hacerlo.

Si un nombre heredado permanece, debe asumirse como excepción temporal y no como centro de identidad.

### REGLA PARA CODEX Y AGENTES

Codex y cualquier agente del proyecto deben operar bajo esta jerarquía nominal:

1. OpenCore = identidad maestra
2. nombres heredados = referencias secundarias o transicionales

No deben introducir nuevos nombres rectores.
No deben reforzar la fragmentación nominal.
No deben tratar nombres heredados como si fueran identidades paralelas equivalentes a OpenCore.

### REGLA DE INTERPRETACIÓN

Si existe discrepancia entre nombres en archivos, estructuras o servicios, debe aplicarse este criterio:

- manda OpenCore como nombre conceptual y rector
- los demás nombres se interpretan como legado o transición
- no se debe ampliar la discrepancia creando nuevas variantes

### REGLA DE MIGRACIÓN NOMINAL

La consolidación de nombres puede realizarse de forma gradual.

No es obligatorio renombrar todo de inmediato, pero sí es obligatorio que toda nueva decisión ya responda a OpenCore como centro nominal.

### OBJETIVO

Evitar fragmentación de identidad, proteger coherencia estructural y asegurar que todos los ángulos del sistema respondan a una misma autoridad nominal.
